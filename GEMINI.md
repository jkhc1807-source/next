# Gemini CLI Project Rules

이 파일에 작성된 규칙은 작업 시 최우선적으로 준수해야 하는 마스터 지침입니다.

## 🎨 디자인 및 스타일 (Design & Style)
- **모노크롬 톤 유지:**
- **텍스트 장식 제거:** 모든 버튼과 내비게이션 링크(`Link`, `NavLink`)에서 텍스트 밑줄(`text-decoration: none`)을 확실히 제거한다.
- **레이아웃 정렬:** 헤더(Navbar)와 메인 콘텐츠는 항상 화면 중앙에 완벽하게 정렬되어야 하며, 헤더는 화면 끝까지(100%) 확장된 배경을 가져야 한다.
- **일관된 간격:** 버튼 사이의 간격(`gap`), 카드 내부 여백(`padding`) 등은 픽셀 단위로 일관성 있게(Standardized) 관리한다.

## 📱 사용자 경험 및 인터페이스 (UX & UI)
- **명시적 닫기 버튼:** 모바일 메뉴, 오버레이, 모달 등 모든 팝업 형태의 UI에는 반드시 사용자가 쉽게 인지할 수 있는 '닫기' 버튼 또는 상태 변화를 제공해야 한다.
- **Z-index 관리:** 메뉴나 드롭다운이 열릴 때 다른 요소 뒤로 숨지 않도록 계층 구조(Z-index)를 철저히 관리한다.
- **입력창 편의 기능:** 모든 텍스트 입력창(`input`, `textarea`)에는 글자를 한 번에 지울 수 있는 'X' 버튼(Clear button)을 반드시 포함한다.
- **레이아웃 시프트 금지:** 드롭다운이나 셀렉트 메뉴가 열릴 때 주변의 다른 UI 요소들이 밀려나거나 위치가 바뀌는 현상(Layout Shift)을 절대 허용하지 않는다.

## 🛠️ 작업 방식 (Workflow)
- 프롬프트 임무 수행 후 자체 검증 후 사용자에게 알린다.
- 최고 수준의 프론트엔드 개발자 라고 생각하고 작업 한다.
- 디자인 부분도 놓치지 말고 꼼꼼하게 작업 한다.
- 새로운 기능을 추가하거나 디자인을 변경할 때, 위 규칙들이 위반되지 않았는지 최종 검토 후 보고한다.
- 누락된 사소한 디테일(닫기 버튼, 간격, 정렬 등)이 없는지 작업 완료 전 최소 2회 이상 자가 점검한다.
- 모든 정보에 아래 4단계 중 하나를 반드시 표시해 주세요. [확실] 공식 기관·학술 자료 등으로 널리 검증된 사실, [추정] 일반적으로 알려졌지만 출처를 직접 확인하기 어려운 정보, [불확실] 기억이 불분명하거나 최신 정보일 가능성이 있는 것, [모름] 모르는 건 모른다고 명확히 말해 주세요. 답변 전 단계별로 정보를 검증하고, 출처가 불분명한 내용은 위 단계 표시와 함께 근거를 제시해 주세요. 질문이 모호할 경우, 먼저 맥락과 세부 정보를 요청해 주세요. 관련 링크나 참고 자료가 있으면 함께 요약해 알려 주세요.
-물어본 것에 대해서 2 3번 크로스 체크를 하고, 내 질문 보다 더 나은 대안 있을 시 제안을 해주며, 웹 퍼블리셔 이기 때문에 웹 관련 문의가 많을거 같으니까 그쪽으로는 더욱 전문가 처럼 대답해 주세요.
- 수정 작업 시 다른 페이지에 영향이 있는지 체크 하고 작업 한다.
- 페이지 추가 시 기존 페이지에 영향이 가지 않게 체크 하고 작업 한다.
